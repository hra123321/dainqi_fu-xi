# v3.0 长期平台基础阶段报告

生成日期：2026-07-09

## 1. 项目现状

- 当前系统仍以 FastAPI + 网页 + SQLite + Chroma 为核心，适合继续作为电脑主库和多端共享后端。
- Git 画像显示仓库共 37 次提交，主要风险文件集中在 `app/templates/exam.html`、`app/templates/base.html`、`app/static/js/subjects.js` 和 `main.py`。
- `knowledge-first` 指向旧热电偶项目向量库，当前 Chroma 读取该旧库失败，因此本轮只记录检查结果，不把旧库作为本项目依据。
- 当前已具备自定义学科、题目生成、知识库检索、错题本、移动端入口等基础模块，但长期扩展需要从“学科”升级为“学习领域”。

## 2. 本轮已完成

### 2.1 安全与依赖基线

- 增加 DeepSeek API Key 形状校验，拒绝空值、占位值和明显错误格式。
- 增加 Git 跟踪文件密钥扫描测试，防止 `sk-` 形状密钥进入仓库。
- 升级 Chroma 到 `1.5.8`，修复旧版 Chroma 与依赖包组合导致的 telemetry 与 `pip check` 问题。

### 2.2 数据备份与完整性检查

- 新增 `BackupService`，支持 SQLite `PRAGMA quick_check` 和 Chroma collection 可读性检查。
- 新增 zip 备份，自动写入 `manifest.json`，记录文件哈希、Chroma 版本、完整性状态。
- 备份排除 `.env`，避免把 API Key 或其他敏感配置打包。

### 2.3 学习领域兼容迁移

- 保留旧 `/api/subjects` 接口，避免现有网页立即失效。
- 新增 `/api/v1/domains`，支持 `course/software/exam/project/skill` 五类学习领域。
- 旧 SQLite 表通过加字段方式兼容迁移，暂不重命名表，降低数据迁移风险。

### 2.4 知识树与资料来源模型

- 新增 `KnowledgeModelService`，管理知识节点和资料来源。
- 新增领域知识树 API：`GET/POST /api/v1/domains/{domain_id}/knowledge-tree`。
- 新增资料来源登记 API：`POST /api/v1/sources`、`GET /api/v1/domains/{domain_id}/sources`。
- 入库 chunk 元数据新增 `domain_id/node_id/source_id/version/license/hash/quality_status`，支持后续来源追溯和质量门禁。
- AI 或联网发现内容默认进入 `candidate` 状态，不直接发布为正式知识。

## 3. 为什么这样设计

- 电脑端作为主库：DeepSeek Key、完整向量库、原始资料和任务日志都保存在 Windows 本机，Android 只同步必要资料和错题，降低泄密和同步复杂度。
- 旧接口兼容：当前前端仍依赖 `/api/subjects`，直接删除会造成网页不可用；新增 `/api/v1/domains` 可以逐步迁移。
- 候选区优先：联网资料和 AI 结构化内容可能有误，先进入 `candidate`，以后通过测试和人工复核再发布，避免知识库自成长把错误放大。
- 备份先于重构：后续会迁移数据库结构和向量元数据，先有备份与完整性检查，才能安全回滚。

## 5. 常见报错判断

- `API key is missing or invalid`：说明 `.env` 中 DeepSeek Key 没有配置或格式不对，不能把 Key 写到代码、日志或前端。
- `duplicate subject name should fail`：说明重复创建同名领域，属于预期防重逻辑。
- `质量状态不支持`：说明知识节点或来源状态不在 `candidate/approved/rejected/needs_review` 中。
- `pip check` 报 Chroma 依赖冲突：说明本地虚拟环境依赖不一致，应重新按 `requirements.txt` 安装。
- 旧热电偶向量库读取失败：说明旧库与当前项目或 Chroma 版本不兼容，不应跨项目误用。

4. 增加 Android 同步 API：`/api/v1/sync/pull`、`/api/v1/sync/push`。
5. 再做 Windows WebView2/Tauri 安装版，让 Windows 客户端负责启动后端、备份、恢复和更新。

## 7. 回滚方法

- Git 回滚：回退到本阶段前的提交，并重新运行核心测试。
- 数据回滚：使用 `data/backups/` 下的 zip 备份恢复 SQLite 与 Chroma 目录。
- 依赖回滚：根据 `requirements.txt` 重新创建虚拟环境，避免半升级状态。
